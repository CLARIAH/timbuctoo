Overview of anansi

Anansi is developed for the Dutch CLARIAH initiative. This initiative sets out to deliver “an e-science infrastructure for the humanities” (as you would probably expect). We have divided the work to achieve this into 5 separate work packages.

  • WP1 is for administrative functions
  • WP3,4 and 5 are for building an infrastructure for a specific research-domain within the humanities
  • WP2 will provide the so called Common Lab. A developer oriented infrastructure for those needs that transcend the individual workpackages.

Next to the commodity infrastructure such as identity management, the main tool that WP2 will build is a tool (called anansi) with a generalized form of the data that is found in the workpackage infrastructures themselves.

The main features that we currently hope to offer using this tool are:

  1. providing Authority lists, lists of data that a researcher can use to bootstrap a research collection such as a diachronous list of places and concepts.
    These lists will be kickstarted by certain handmade collection but we hope to devise ways of expanding on them by crowdsourcing submitted data.
  2. Being able to execute queries on the total integrated set of research data. Even being able to only query at a very abstract level (in what collections is Person X mentioned) is probably very valuable.
  3. Being able to build interfaces that help data entry, by presenting similar entities from the database (aka “did you mean…”)

(This list is of course neither exhaustive, not exclusive. But instead serves as an illustration of the use cases we envision)

Overview of timbuctoo

Timbuctoo’s development precedes that of anansi and parts of anansi are designed based upon the lessons learned while developing timbuctoo. Timbuctoo aims to provide the following features (also within the domain of the humantities):

  1. A low-maintenance store for structured research data. It should be able to contain data for many different projects. It is hoped that this will lower the treshhold (budget-wise) for researchers to make use of IT tools for analysing data.
  2. A foundation for building user interfaces that are easy to use for humanities researchers and can be cheaply implemented for new datasets (initially we propose to build a faceted search and an edit and details screen)
  3. Support users by modelling their data. We propose to do this by providing sane defaults.
  4. Increase the interoperability of researches by offering a sane set of defaults.

Incidental constraints

The following constraints are not directly derived from the project’s goal, but

  1. There needs to be a way of identifying the provenance of a value. A user of the database needs to be able to determine, after a fashion, how a value entered the database.
  2. It is assumed that the research questions that the databases need to answer are more about relations between entities than about a lookup of all entities containing a specific set of attributes. We’re assuming, based on previous experience, that we need a so called “graph database” to be able to easily keep up the performance while implementing the queries that will be expected of us.
  3. All data in the timbuctoo and anansi will be publicly visible (our employer, the Royal Academy of Sciences, requires all research data to be public data). The data must not be publicly editable.
  4. All data in anansi must be derived data. Anansi is not a data store, but at most a data cache.

Target user

Based on the goals we can state that our target user is a developer of research applications or scientific programmer. We will therefore target our documentation and interfaces towards these groups.

Throughout this document, the term user might refer to the developers using our software and the researchers using their software. Usually it will be evident from context which group is referred to. When needed we will make the distinction more explicit by refering to the developers of research applications as “developer-users” and the researchers as “end-users”. Research tools is the generic term used to refer to the variety of software applications and one-off interactions that use our infrastructure.

As can be seen in the overview the needs of the end-users are still the drivers for the features of our system. Moreover, we will keep the needs of the research patrons in mind. The needs of other groups and stakeholders will not be actively considered.

The application landscape

Each workpackage, and timbuctoo, will be able to present their information encoded in the RDF datamodel (exact encodings have not yet been specified). The anansi crawler will read this information and stores it in the anansi database. This database can thus be deleted and re-generated at will by re-harvesting from the providing data sources. Though we will probably implement a protocol that allows for efficient updating when both sides are still synchronized. The anansi database will only contain the most recent version of the data any versioning information is expected to be stores at the workpackages themselves.

With this in place we have one, big, set of data that is probably not very linked at all. We will then add automated or manual reasonings to extract make the data better linked. The results of these reasonings are conceptually separate from the anansi database. They can the result of a researcher annotating data using a GUI, or an OWL inferencer or something else.

The actual features will then use those two databases to construct their own dataset. A developer will then use one such feature.

(A quick note regarding technologies: so far I’ve been looking at the usual suspects for data storage and leaning towards titan with a. hbase backend. Though I recently heard good things about stardog.)

Current implementation

At this moment, only timbuctoo has an actual implementation. It is built as a java web-application that used neo4j and solr for data storage. The data model as used by timbuctoo is described below. The API is documented in the interface documentation.

The data model

The current data model is based on the lowest common denominator of the data abstraction layer (tinkerpop2) and the database (neo4j). This means that there are:

  • Vertices (what neo4 calls nodes)
  • Properties
  • Edges (a relation between two vertices)

Vertices represent a version of an entity. Properties represent the values of that entity that do not warrant their own identifier. Edges represent the relations between vertices and might have properties of their own (though this happens rarely)

Versioning

(A, B, C & D) The light blue square represents a vertex and thus one version of the entity. In A and B there is only 1 versie of Maria and Pieter, in C there are 2 versions of Maria and 1 of Pieter and in D there are 2 versies of both Maria and Pieter. The set of vertices with the same value for the property tim_id form a singly linked list through edges of the type isVersionOf. These are grouped in a darker blue square and labeled “version series”

(B) When an edge is created between two vertices, no new version is created. Both the vertex and the relations have a created_at property so that one can identify that there was a time when the vertices did exist, but the relation did not.

(C & D) When a property is changed a new version _is_ created. The relations to the old vertex are deleted and new relations are created using the new vertex. These new relations keep their created_at date, so it is possible to find all the vertices that existed while the old vertex was active by following the isVersionOf relation up to the latest version and iterating over its relations.

(Not in the drawing) When a property of a relation is modified a new relation is created and the old one marked as obsolete. similar to vertices.

(not in th drawing) The drawing slightly misrepresents the actual workings of the data persistence code in order to make it easier to understand. In reality the duplication of the object does not happen right before a modification, but directly after a “persistent id” has been received from the handle server. The race conditions that this might introduce are prevented by disallowing writes depending on whether a persistent id is available or not. (There are three states possible: the property is non-existent, the property exists but has the value NULL and the property exists)

Variants

The current timbuctoo also code to achieve most of the goals of anansi. These goals are implemented by storing all the data of the different research projects on a certain entity in the same vertex using namespaces for the properties.

There is also 1 namespace that contains the “abstract” data that does have meaning within the domain, such as a persons name. De facto this is the data of the first research environment that entered a value for that property. This abstract-level data is currently unused.

There is also data that is not domain related, such as the isLatest property. This is again placed in its own namespace.

Java code

In de java code wordt een vertex geserialiseerd naar een java object van een specifieke class. Indien geen versie wordt opgegeven is dit altijd de head van de versie-serie.

De java classes worden door de timbuctoo developers geschreven en hierbij wordt gebruikt gemaakt van java’s mogelijkheden voor inheritance en static typing (de velden liggen vast, een subclass moet alle velden hebben die een superclass heeft). Inheritance wordt vooral gebruikt om de doelen 3&4 van timbuctoo te ondersteunen. Ieder object in een VRE dient ook een representatie als abstract type te hebben.

Binnen een java class zijn alleen de eigenschappen beschikbaar die bij die VRE horen. Daarnaast worden utility functies en data vertalingen in de modellen uitgewerkt. Alsmede een aantal methodes ten behoeve van de frontend, zoals displayName, getClientRepresentation en createRelSearchRep. De java classe bepaalt ook hoe, en onder welke naam een property geindexeerd wordt en bepaald daarmee mede de user interface van de faceted search.

Benoemde problemen (timbuctoo)

Modellen via java classes

We kunnen ons een aantal voordelen bedenken die static typing en java-inheritance aan de gebruiker biedt:

  1. Door een standaard base class te maken worden andere implementaties gelijkvormiger. Waarmee de mogelijkheid tot interoperabiliteit tussen VRE’s toeneemt
  2. De code die de gebruiker schrijft kan er vanuit gaan dat een bepaald type altijd bepaalde eigenschappen heeft en dat die eigenschappen er altijd uitzien als…

Er zijn ook een tweetal nadelen:

  1. Het toevoegen van een property vereist een wijziging in de source code en een redeploy van de applicatie. Hierdoor worden bepaalde properties niet toegevoegd omdat de eindgebruiker dit niet meer van de ontwikkelafdeling mag vragen.
  2. Niet alle objecten in een VRE hebben een duidelijke relatie met een van de abstracte supertypes in de algemene collectie. Een nieuw type toevoegen vereist dat een supertype bedacht wordt dat van toepassing is op huidige en toekomstige VRE’s.  In de praktijk worden nieuwe typen daarom niet toegevoegd, maar in plaats daarvan via een workaround gemodelleerd
  1. publish event is niet gemodelleerd in women writers
  2. charterdocument laat de titel en de creator property altijd leeg en schend daarmee niet de letter van het contract maar wel de intentie.

Versioning in neo4j

Op dit moment is het erg lastig/omslachtig om een query te schrijven die gebruik maakt van de versie-awareness van de database.

Relaties vs properties

Relaties en properties zijn in de HTTP API en in de user interface erg verschillend. Ze zouden gelijkvormiger kunnen worden. (Een relatie kan een property zijn die als waarde een vertex heeft)

Time aware queries

Hoewel er vanuit het domein bepaalde properties met tijd kunnen worden opgeslagen en daarop gequeried kan worden is het nog niet mogelijk om in het algemeen een time-aware query te doen. (Query x, maar dan nu alleen voor de jaren 20)
